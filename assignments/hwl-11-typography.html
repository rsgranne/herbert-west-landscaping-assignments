Default font

In the first CSS assignment, we set the font-family for the html selector to this: 

"Source Sans 3", Verdana, Geneva, Tahoma, sans-serif

There is nothing per se wrong about that font stack, but there are a few issues.

First of all, I know I have Source Sans 3 installed, but I’m almost certain virtually no one else does. So that means hardly anyone will see the ideal font I’d like to use on this website.

Second, there is a way to have your computers download & use the Source Sans 3 font on your computer using techniques discussed later in CSS Typography. However, that means an extra download, which slows down the page load time. Bummer.

In the last decade, many Web developers have moved toward the idea of using native font stacks as a way to solve these problems. Basically, instead of linking to specific fonts, you instead tell the browser’s rendering engine to use the default operating system user interface font for your device.

  • If you use a Mac with a version of macOS since 10.11 El Capitan (which rolled out in 2015!), then you are looking at Apple’s San Francisco typeface in menus, the Finder, the Desktop, & basically anywhere on your Mac that is part of macOS (San Francisco has also been the default on iOS since version 9, also released in 2015).
  • If you use a PC with a version of Windows from Vista (also available since 2007) on up to today, then you constantly see a typeface named Segoe UI in the user interface.
  • Most Android phones have used Roboto since 2011, when it first appeared on Android 4.0 “Ice Cream Sandwich”.

With that in mind, here is the native font stack used by Bootstrap, which we will adopt today for the default font on the Herbert West Landscaping website (we will continue to use a serif-based font stack for headings, however): 

system-ui, -apple-system, "Segoe UI", Roboto, "Helvetica Neue", Arial, "Noto Sans", "Liberation Sans", sans-serif, "Apple Color Emoji", "Segoe UI Emoji", "Segoe UI Symbol", "Noto Color Emoji"

Remember, the browser’s rendering engine will start with the first specified font & go down the list until if gets to one it supports. Let’s break that font stack down so we understand why Bootstrap chose those fonts, with each operating system listed after each font.

  • system-ui: This has come to mean “the default operating system font” for all modern browsers
  • -apple-system: Apple
  • "Segoe UI": Windows
  • Roboto: Android
  • "Helvetica Neue": Older Apple devices
  • "Noto Sans": Linux
  • "Liberation Sans": Linux
  • Arial: Basic fallback for almost all operating systems
  • sans-serif: Generic family fallback
  • "Apple Color Emoji": Apple
  • "Segoe UI Emoji": Windows
  • "Segoe UI Symbol": Windows
  • "Noto Color Emoji": Linux

Set the font-family for <html> to the system font stack above. Save it & take a look. What do you think?

Now the problem is that the multi-column rule is unnecessary & looks too cluttered, given the borders around .col. We need to get rid of that rule. However, when we styled .multi-column in the previous assignment (around all the colors on landscape-maintenance.html), we wanted the rule there, but now we don’t. We can’t just remove column-rule: 2px dotted hsl(150, 70%, 31%); from .multi-column, since we want the rule to appear most of the time when we use that class. So what’s the solution?

The answer is to add the no-rule class to <div class="multi-column"> so that it looks like this: <div class="multi-column no-rule">.

Then, in main.css, add a compound selector named .multi-column.no-rule. Remember, that will match any element with both the multi-column class and the no-rule class. Now, to override the column rule we already have in place for .multi-column, put this declaration in place for our new compound selector: column-rule-width: 0;. One last thing: you might notice that a <div class="col"> wraps from the first into the second column, like this:

To fix this, add display: inline-block; to the .multi-column > .col selector. Reload the page, & you should see something like this:

Why use inline-block with display? You know what display: block does, & you know what display: inline does. When you use inline-block, you tell the rendering engine to display the element inline with other elements, but also apply some block-level properties to it, such as a fixed width or height, padding or margins, or borders.

When you apply display: inline-block; in this instance, the block part prevents the <div> from wrapping across into a new column, which is exactly what we want.
